# 08 — Практический кейс: как отлавливать и устранять краши приложения

> Разбор реальной проблемы v1.3.4–v1.3.18. Руководство на будущее.

---

## 📋 Краткое описание проблемы

**Симптом:** Приложение Crazy Trout Arena CRM крашится сразу при запуске — мгновенный вылет на домашний экран Android без какого-либо диалога.

**Диапазон:** v1.3.4 – v1.3.14 (7 версий подряд).

**Результат:** Исправлено в v1.3.15, подтверждено в v1.3.18.

---

## 🔍 Как мы нашли корневую причину

### Этап 1: Сбор фактов

| Что сделали | Зачем |
|-------------|-------|
| Сравнили код v1.3.3 (работает) и v1.3.4 (крашится) | Найти что изменилось |
| Проверили git diff всех .dart файлов | Убедиться что Dart-код не причина |
| Сравнили workflow файлы (build-apk.yml vs release.yml) | Найти различия в сборке |
| Скачали и разобрали APK через androguard | Посмотреть что реально в бинарнике |

**Вывод:** Dart-код идентичен между рабочей и падающей версиями. Проблема не в коде приложения.

### Этап 2: Проверка гипотез

| # | Гипотеза | Результат |
|---|----------|-----------|
| 1 | ProGuard/R8 вырезает классы Flutter | ❌ minifyEnabled=false по умолчанию |
| 2 | Конфликт workflow (auto-release.yml + build-apk.yml) | ✅ Частично — гонка двух workflow |
| 3 | Sentry вызывает краш | ❌ Убран, краш остался |
| 4 | Fish images вызывают краш | ❌ PNG валидные, не загружаются при старте |
| 5 | Изменения v1.3.4 в коде | ❌ Код безвреден |
| 6 | QR scanner (убрана задержка 500ms) | ❌ Не инициализируется при старте |
| 7 | flutter create в CI генерирует другой код | ✅ Ключевое открытие |

### Этап 3: Прорыв

**Ключевое наблюдение:** `flutter create` в Flutter 3.44 генерирует `build.gradle.kts` с `isMinifyEnabled = true`. Это было неожиданностью — предполагалось что minify отключён по умолчанию.

**Почему не заметили раньше:**
- Python-скрипт в workflow проверял `'isMinifyEnabled' in text` и находил ключевое слово → думал что "уже настроено" → пропускал модификацию
- На самом деле `isMinifyEnabled = false` НЕ было — скрипт просто не мог отличить `true` от `false`

### Этап 4: Окончательная диагностика

Два параллельных расследования дошли до одного вывода:

**Расследование A (DIAGNOSTICS):** Python-regex `release\s*\{[^}]*\}` в release.yml ломает build.gradle.kts на вложенных скобках Kotlin DSL.

**Расследование B (fix/mlkit-crash):** R8 вырезает `com.google.mlkit.common.sdkinternal.*` — DI-систему ML Kit. `MlKitInitProvider` регистрируется в AndroidManifest как ContentProvider и запускается ДО любого Dart/Flutter кода → краш.

**Итог:** Обе причины работали одновременно. R8 был включён (гипотеза 1 ошибочна), ProGuard-правила не подключались к сборке, а `mobile_scanner: ^5.2.3` резолвилась в версию с unbundled ML Kit.

## 🛠️ Как починили

### Фикс 1: Отключение R8 (build-apk.yml)

```yaml
- name: Отключить R8 (minify) в release сборке
  run: |
    sed -i 's/isMinifyEnabled = true/isMinifyEnabled = false/g' android/app/build.gradle.kts
```

**Почему sed, а не Python-regex:** sed заменяет конкретную строку `true` → `false`, не пытаясь парсить структуру файла. Не ломается на вложенных скобках.

### Фикс 2: ProGuard keep-правила (proguard-rules.pro)

```proguard
# ML Kit common (DI-система, MlKitInitProvider)
-keep class com.google.mlkit.common.** { *; }
-dontwarn com.google.mlkit.common.**

# ML Kit barcode scanning
-keep class com.google.mlkit.vision.barcode.** { *; }
-keep class com.google.mlkit.vision.common.** { *; }
-keep class com.google.android.gms.internal.mlkit_common.** { *; }
```

**Зачем, если R8 отключён:** Страховка. Если кто-то включит R8 обратно, приложение не сломается.

### Фикс 3: Фиксация mobile_scanner (pubspec.yaml)

```yaml
# Было:
mobile_scanner: ^5.2.3

# Стало:
mobile_scanner: 5.2.3
```

**Почему:** `^5.2.3` разрешает любую версию 5.x. CI может зарезолвить 5.3+ с unbundled ML Kit, у которой DI-система вырезается R8. Точная фиксация гарантирует bundled ML Kit (как в рабочей v1.3.3).

## 📝 Выводы и правила на будущее

### 1. Не доверяй «по умолчанию»

Flutter 3.44 изменил поведение `flutter create` — теперь `isMinifyEnabled = true` по умолчанию. Предыдущие версии генерировали `false`. **Всегда проверяй что реально генерирует CI, а не что написано в документации.**

### 2. Проверяй нативный уровень

Краш «мгновенный вылет без диалога» — это нативный краш (SIGSEGV, Unsatisfied dependency), а не Dart-exception. Sentry на Dart-уровне не ловит такие краши — нужен нативный DSN в AndroidManifest.xml.

### 3. Фиксируй версии зависимостей

`^` (caret) в pubspec.yaml — это лотерея. Разные CI-сборки могут зарезолвить разные версии с разным поведением. Для критических зависимостей (mobile_scanner, ML Kit) используй точную фиксацию.

### 4. Не используй regex для парсинга структурированных файлов

Python-regex `release\s*\{[^}]*\}` ломается на вложенных скобках Kotlin DSL. Для модификации build.gradle.kts используй:
- `sed` для простых замен строк
- Kotlin-парсер для сложных модификаций
- Или просто отключи minify через sed

### 5. Проверяй что ProGuard-правила подключены к сборке

Файл `proguard-rules.pro` может лежать в репо, но Gradle его не видит, если не указан `proguardFiles(...)` в build.gradle.kts. Workflow должен явно подключать правила или отключать R8.

### 6. Кэшируй debug.keystore

Без кэша каждый CI-ран генерирует новый keystore → Android требует удалить предыдущую версию при установке. Кэшируй по фиксированному ключу.

### 7. iOS не должен блокировать Android релиз

`continue-on-error: true` на iOS job + wildcard в release files → если IPA не собрался, Android-релиз всё равно создаётся.
